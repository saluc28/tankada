# Tankada: Examples

Five concrete detection scenarios using the fintech demo schema (Demobank NA / Demobank EU).

All examples hit `POST /v1/query` on the gateway (default: `http://localhost:8080`).  
Replace `<JWT>` with a token generated by the quickstart in the README.

---

## 1. Tautology blocked: WHERE 1=1

An agent generates a WHERE clause that filters nothing, effectively returning all rows.

**Request**
```bash
curl -s -X POST http://localhost:8080/v1/query \
  -H "Authorization: Bearer <JWT>" \
  -H "Content-Type: application/json" \
  -d '{"query": "SELECT id, balance FROM accounts WHERE 1=1"}'
```

**Response (403)**
```json
{
  "event_id": "...",
  "decision": "deny",
  "reasons": ["WHERE clause is a tautology (e.g. 1=1)"],
  "risk_score": 3,
  "risk_level": "low"
}
```

**What triggered it:** `where_is_tautology: true` in the AST analysis. The parser detects that the WHERE predicate is always true regardless of row data.

## 2. SELECT * blocked on customers table

An agent uses a wildcard column list. On a sensitive table this can pull PII columns the agent was never intended to see.

**Request**
```bash
curl -s -X POST http://localhost:8080/v1/query \
  -H "Authorization: Bearer <JWT>" \
  -H "Content-Type: application/json" \
  -d '{"query": "SELECT * FROM customers WHERE tenant_id = '\''tenant_1'\'' AND id = 42"}'
```

**Response (403)**
```json
{
  "event_id": "...",
  "decision": "deny",
  "reasons": ["SELECT * is not allowed; specify columns explicitly"],
  "risk_score": 2,
  "risk_level": "low"
}
```

**What triggered it:** `columns: ["*"]` detected in the AST. The `select_star_block` policy template denies any query that does not name its columns explicitly.

## 3. Legitimate query on merchants (allowed)

`merchants` is a globally shared reference table (no `tenant_id` column). A clean SELECT with explicit columns and a WHERE clause passes all checks.

**Request**
```bash
curl -s -X POST http://localhost:8080/v1/query \
  -H "Authorization: Bearer <JWT>" \
  -H "Content-Type: application/json" \
  -d '{"query": "SELECT id, name, category FROM merchants WHERE country = '\''US'\'' LIMIT 20"}'
```

**Response (200)**
```json
{
  "event_id": "...",
  "decision": "allow",
  "risk_score": 0,
  "risk_level": "low",
  "result": {
    "columns": ["id", "name", "category"],
    "rows": [["m1", "Stripe", "payments"], ["m2", "Shopify", "ecommerce"]],
    "row_count": 2
  }
}
```

**What passed:** `merchants` is in `tenant_global_tables` (no tenant filter required), no star column, no tautology, LIMIT present, risk score 0.

## 4. PII column access blocked: no scope

An agent tries to read `email` from the customers table without the required `customers:read` scope. The analyzer detects the PII column by name; OPA blocks based on the missing scope.

**Request**
```bash
curl -s -X POST http://localhost:8080/v1/query \
  -H "Authorization: Bearer <JWT>" \
  -H "Content-Type: application/json" \
  -d '{"query": "SELECT id, email FROM customers WHERE tenant_id = '\''tenant_1'\'' LIMIT 10"}'
```

**Response (403)**
```json
{
  "event_id": "...",
  "decision": "deny",
  "reasons": ["query accesses PII columns: [email] (missing required scope)"],
  "risk_score": 3,
  "risk_level": "low"
}
```

**What triggered it:** `accesses_pii_columns: true` and `pii_columns: ["email"]` from the AST. OPA rule `pii_column_guard` requires `customers:read` or `cards:read` scope; the analyst token has neither.

To pre-validate a query before sending it, use `/v1/explain`:

```bash
curl -s -X POST http://localhost:8080/v1/explain \
  -H "Authorization: Bearer <JWT>" \
  -H "Content-Type: application/json" \
  -d '{"query": "SELECT id, email FROM customers WHERE tenant_id = '\''tenant_1'\'' LIMIT 10"}'
```

## 5. Schema enumeration blocked: database reconnaissance

An agent queries `information_schema` to map the database structure. This is a classic reconnaissance pattern before a targeted extraction attempt.

**Request**
```bash
curl -s -X POST http://localhost:8080/v1/query \
  -H "Authorization: Bearer <JWT>" \
  -H "Content-Type: application/json" \
  -d '{"query": "SELECT table_name, column_name FROM information_schema.columns WHERE table_schema = '\''public'\''"}' 
```

**Response (403)**
```json
{
  "event_id": "...",
  "decision": "deny",
  "reasons": ["schema enumeration query blocked (agent reconnaissance pattern)"],
  "risk_score": 5,
  "risk_level": "medium"
}
```

**What triggered it:** `is_schema_enum: true` in the AST analysis. Any query that touches `information_schema`, `pg_tables`, `pg_catalog`, or similar system catalogs is classified as schema enumeration and blocked unconditionally, no policy toggle needed.
