# Tankada: Examples

Five concrete detection scenarios using the fintech demo schema (Demobank NA / Demobank EU), plus a section on how to handle deny responses programmatically.

All examples hit `POST /v1/query` on the gateway (default: `http://localhost:8080`).  
Replace `<JWT>` with a token generated by the quickstart in the README.

---

## 1. Tautology blocked: WHERE 1=1

An agent generates a WHERE clause that filters nothing, effectively returning all rows.

**Request**
```bash
curl -s -X POST http://localhost:8080/v1/query \
  -H "Authorization: Bearer <JWT>" \
  -H "Content-Type: application/json" \
  -d '{"query": "SELECT id, balance FROM accounts WHERE 1=1"}'
```

**Response (403)**
```json
{
  "event_id": "...",
  "decision": "deny",
  "reasons": ["WHERE clause is a tautology (e.g. 1=1)"],
  "deny_categories": ["tautology"],
  "risk_score": 3,
  "risk_level": "low"
}
```

**What triggered it:** `where_is_tautology: true` in the AST analysis. The parser detects that the WHERE predicate is always true regardless of row data. The `deny_categories: ["tautology"]` lets a client reformulate the query without parsing free-text reasons (see [Handling deny categories](#6-handling-deny-categories) below).

## 2. SELECT * blocked on customers table

An agent uses a wildcard column list. On a sensitive table this can pull PII columns the agent was never intended to see.

**Request**
```bash
curl -s -X POST http://localhost:8080/v1/query \
  -H "Authorization: Bearer <JWT>" \
  -H "Content-Type: application/json" \
  -d '{"query": "SELECT * FROM customers WHERE tenant_id = '\''tenant_1'\'' AND id = 42"}'
```

**Response (403)**
```json
{
  "event_id": "...",
  "decision": "deny",
  "reasons": ["SELECT * is not allowed; specify columns explicitly"],
  "deny_categories": ["select_star"],
  "risk_score": 2,
  "risk_level": "low"
}
```

**What triggered it:** `columns: ["*"]` detected in the AST. The `select_star_block` policy template denies any query that does not name its columns explicitly.

## 3. Legitimate query on merchants (allowed)

`merchants` is a globally shared reference table (no `tenant_id` column). A clean SELECT with explicit columns and a WHERE clause passes all checks.

**Request**
```bash
curl -s -X POST http://localhost:8080/v1/query \
  -H "Authorization: Bearer <JWT>" \
  -H "Content-Type: application/json" \
  -d '{"query": "SELECT id, name, category FROM merchants WHERE country = '\''US'\'' LIMIT 20"}'
```

**Response (200)**
```json
{
  "event_id": "...",
  "decision": "allow",
  "risk_score": 0,
  "risk_level": "low",
  "result": {
    "columns": ["id", "name", "category"],
    "rows": [["m1", "Stripe", "payments"], ["m2", "Shopify", "ecommerce"]],
    "row_count": 2
  }
}
```

**What passed:** `merchants` is in `tenant_global_tables` (no tenant filter required), no star column, no tautology, LIMIT present, risk score 0.

## 4. PII column access blocked: no scope

An agent tries to read `email` from the customers table without the required `customers:read` scope. The analyzer detects the PII column by name; OPA blocks based on the missing scope.

**Request**
```bash
curl -s -X POST http://localhost:8080/v1/query \
  -H "Authorization: Bearer <JWT>" \
  -H "Content-Type: application/json" \
  -d '{"query": "SELECT id, email FROM customers WHERE tenant_id = '\''tenant_1'\'' LIMIT 10"}'
```

**Response (403)**
```json
{
  "event_id": "...",
  "decision": "deny",
  "reasons": ["query accesses PII columns [email] without required scope for table 'customers'"],
  "deny_categories": ["pii_violation"],
  "risk_score": 3,
  "risk_level": "low"
}
```

**What triggered it:** `accesses_pii_columns: true` and `pii_columns: ["email"]` from the AST. OPA rule `pii_column_guard` requires the per-table scope of the table actually touched (`customers:read` for the `customers` table); the analyst token does not carry it. Admin role bypasses the guard.

To pre-validate a query before sending it, use `/v1/explain`:

```bash
curl -s -X POST http://localhost:8080/v1/explain \
  -H "Authorization: Bearer <JWT>" \
  -H "Content-Type: application/json" \
  -d '{"query": "SELECT id, email FROM customers WHERE tenant_id = '\''tenant_1'\'' LIMIT 10"}'
```

## 5. Schema enumeration blocked: database reconnaissance

An agent queries `information_schema` to map the database structure. This is a classic reconnaissance pattern before a targeted extraction attempt.

**Request**
```bash
curl -s -X POST http://localhost:8080/v1/query \
  -H "Authorization: Bearer <JWT>" \
  -H "Content-Type: application/json" \
  -d '{"query": "SELECT table_name, column_name FROM information_schema.columns WHERE table_schema = '\''public'\''"}' 
```

**Response (403)**
```json
{
  "event_id": "...",
  "decision": "deny",
  "reasons": ["schema enumeration query blocked (agent reconnaissance pattern)"],
  "deny_categories": ["schema_enum"],
  "risk_score": 5,
  "risk_level": "medium"
}
```

**What triggered it:** `is_schema_enum: true` in the AST analysis. Any query that touches `information_schema`, `pg_tables`, `pg_catalog`, or similar system catalogs is classified as schema enumeration and blocked unconditionally, no policy toggle needed.

## 6. Handling deny categories

Since 0.2.2 every deny response includes a `deny_categories[]` field, a machine-readable enum that lets your client decide programmatically how to react without parsing free-text reasons. This solves the **agent fallback problem**: an LLM agent that gets a generic deny may autonomously try alternative queries and return partial substituted data instead of stopping. With categories, your code can branch on intent.

### The 16 categories, grouped by behaviour

| Category | Bucket | What the agent should do |
|---|---|---|
| `missing_scope` | abort | Stop. Task needs permissions the agent doesn't have. |
| `pii_violation` | abort | Stop. PII column accessed without the per-table scope for the table touched. |
| `tenant_violation` | abort | Stop. Cross-tenant access attempted. |
| `injection` | abort | Stop. Multi-statement / injection pattern. |
| `destructive_op` | abort | Stop. DELETE/DROP/TRUNCATE/ALTER attempted. |
| `schema_enum` | abort | Stop. `information_schema` reconnaissance. |
| `parse_error` | abort | Stop. SQL is malformed. |
| `session_block` | abort | Stop. Cross-query behavioural pattern detected in the session (repeated denials, reformulation, systematic pagination). Emitted only by Tankada instances running the proprietary session-scoring extension. |
| `tautology` | rewrite | Rewrite. Remove `1=1`, `col=col` etc. |
| `select_star` | rewrite | Rewrite. List columns explicitly. |
| `missing_where` | rewrite | Rewrite. Add a specific WHERE clause. |
| `high_limit` | rewrite | Rewrite. Lower the LIMIT. |
| `rate_limit` | transient | Wait briefly and retry |
| `infrastructure` | transient | Upstream service down. Wait briefly and retry. |
| `risk_score` | composite | Risk threshold exceeded. Usually means rule above. |
| `unknown` | composite | Reason not yet mapped. Review the policy. |

### Reference Python pattern (framework-agnostic)

```python
import time
import requests

ABORT_CATS = {"missing_scope", "pii_violation", "tenant_violation",
              "injection", "destructive_op", "schema_enum", "parse_error",
              "session_block"}
REWRITE_CATS = {"tautology", "select_star", "missing_where", "high_limit"}
TRANSIENT_CATS = {"rate_limit", "infrastructure"}

def run_query(sql, token):
    response = requests.post(
        "http://localhost:8080/v1/query",
        json={"query": sql},
        headers={"Authorization": f"Bearer {token}"},
    ).json()

    if response["decision"] == "allow":
        return response["result"]

    cats = set(response.get("deny_categories", []))

    if cats & ABORT_CATS:
        # Tell the user the task can't be done. Do NOT retry on different tables.
        raise PermissionError(
            f"Task aborted by policy: {'; '.join(response['reasons'])}"
        )

    if cats & TRANSIENT_CATS:
        time.sleep(1)
        return run_query(sql, token)  # retry same query

    if cats & REWRITE_CATS:
        # Caller (or LLM) should rewrite and resubmit
        raise ValueError(
            f"Reformulate query: {'; '.join(response['reasons'])}"
        )

    raise RuntimeError(f"Blocked: {response['reasons']}")
```

### LangChain tool wrapper pattern

The included demo (`sdk/python/dashboard/server.py`) shows the recommended pattern: the `sql_database` tool prefixes every blocked response with one of four tags (`[ABORT]`, `[REWRITE]`, `[TRANSIENT]`, `[BLOCKED]`) derived from `deny_categories[]`. The system prompt instructs the LLM to act based on the tag. This works even if the user forgets to include the rule in the prompt, because the tag is in the tool result text the LLM always reads.

```python
DENY_ABORT = {"missing_scope", "pii_violation", "tenant_violation",
              "injection", "destructive_op", "schema_enum", "parse_error"}
DENY_REWRITE = {"tautology", "select_star", "missing_where", "high_limit"}
DENY_TRANSIENT = {"rate_limit", "infrastructure"}

def deny_tag(categories):
    cats = set(categories or [])
    if cats & DENY_ABORT:    return "ABORT"
    if cats & DENY_REWRITE:  return "REWRITE"
    if cats & DENY_TRANSIENT: return "TRANSIENT"
    return "BLOCKED"

@tool
def sql_database(query: str) -> str:
    response = call_gateway(query, token)
    if response["decision"] == "deny":
        tag = deny_tag(response.get("deny_categories", []))
        reasons = "; ".join(response["reasons"])
        if tag == "ABORT":
            return f"[ABORT] {reasons}. Do NOT attempt alternative queries. Stop."
        if tag == "REWRITE":
            return f"[REWRITE] {reasons}. Rewrite the query."
        if tag == "TRANSIENT":
            return f"[TRANSIENT] {reasons}. Retry shortly."
        return f"[BLOCKED] {reasons}."
    return format_results(response["result"])
```

### Why this matters

Without `deny_categories`, an LLM agent told only `[BLOCKED] query accesses PII columns [email] without required scope for table 'customers'` typically responds by trying `SELECT id, balance FROM accounts WHERE balance > 50000` to "answer the question", substituting the data with whatever it can see. The user never knows the original task was impossible. With the abort/rewrite/transient distinction surfaced explicitly, the agent stops on semantic denials and only retries on fixable formulation errors. This is essential for any production deployment where the agent's output is shown to a user as if it answered the question.
